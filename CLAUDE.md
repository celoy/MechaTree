# MechaTree — guide for Claude sessions

## What this project is

MechaTree is a Python (with Cython + C++) port of a Fortran90 simulator of **tree growth in response to wind and light**. The model captures self-pruning, allometric scaling, and self-similar tree architectures that emerge from the mechanical and photosynthetic constraints. It accompanies:

> Eloy, C., Fournier, M., Lacointe, A. et al. *Wind loads and competition for light sculpt trees into self-similar structures.* **Nat Commun 8, 1014 (2017).** https://doi.org/10.1038/s41467-017-00995-6

## Who this is for

The intended end user is a **scientist working in forestry**, or someone adopting MechaTree for a **course or research project**. They are not assumed to know Fortran or C++.

API implications:
- Pythonic identifiers in the public surface; cross-reference Fortran names in docstrings/comments only.
- Errors over silent sentinels (raise `IndexError`/`ValueError`, don't return `-1`).
- Examples in `examples/` should read like recipes — runnable end-to-end with one command, minimal boilerplate.
- Config in YAML/JSON, not hand-edited Python.

## Design principles for the simulator port

These principles came out of the user's roadmap notes. They guide all Step-8+ work; revisit before re-litigating.

- **Composition over inheritance.** A tree is configured with a list of "functionalities" (mechanics, light response, growth law, pruning rule) chosen at construction. Avoid a single giant `PyTree` subclass; prefer pluggable hooks.
- **Mechanics is central.** Load propagation (wind force → bending moments → stresses up the tree graph) lives in the compiled core. Hot path; do not push to Python.
- **Light interception is detached.** Separate module. Input: a list of `Leaf` records (location, diameter, transparency). Output: light absorbed per leaf. Decoupled from `PyTree` so it can apply to a single tree, a forest, or any leaf cloud.
- **Genome as callback.** Fortran's `neural_branch` / `neural_reserve` decision functions become user-supplied Python callables (or a small set of named built-ins). Trees do not own a genome class; they accept decision callables.
- **Evolution lives behind a clean boundary.** The evolutionary algorithm (`EvoluAlgo.f90`, `mod_evolu.f90`) gets ported as part of Step 21. Whether it ships as `mechatree.evolution` or as a sister package is one of Step 21's open design questions; either way it must import the simulator, never the other way round.
- **Forest is a container.** Many trees, spatial layout, lifecycle (birth → growth → death). Cross-tree light competition handled by the detached light module operating on the union of leaves.
- **Config in YAML/JSON.** Replace Fortran `.ini` files. Schema validated; an example file ships with the package.

### Step 19 notes (closed 2026-05-25)

Per-cell light interception is now `light = tau ** depth`, where `depth` is each leaf's 0-indexed rank in Z'-descending order within its shadow cell (topmost leaf has depth 0).
- **Algorithm**: [src/mechatree/light/interception.py](src/mechatree/light/interception.py) computes per-leaf depth in one vectorised pass — `np.unique(return_index=True)` on the lexsort-sorted cell keys gives each group's start, `np.repeat` broadcasts that back, and `depth = arange(n) - group_start_sorted`. The final write is `leaves.light_per_direction[:, k] = leaf_transparency ** depth`. NumPy follows IEEE `0**0 = 1, 0**k = 0` for `k >= 1`, so `tau = 0` recovers the previous binary topmost-wins behaviour exactly with no special-case branch.
- **Default `tau = 0.5`** (Eloy et al., Nat Commun 2017). This is a behavioural change vs. the prior binary default: understorey leaves now contribute photosynthate they didn't before, so quantitative tree size / leaf count under the default config shifts. Structural tests (reproducibility under seed, callback firing, etc.) are unaffected and still pass.
- **YAML / API**: [`LightConfig.leaf_transparency`](src/mechatree/config.py) is the canonical knob, surfaced via the YAML `light:` block. Validated to `[0, 1]` in `LightConfig.__post_init__` and again in `intercept()`. Threaded through [`mechatree.simulate.grow_tree`](src/mechatree/simulate.py) and [`mechatree.forest.Forest.step`](src/mechatree/forest.py); [`mechatree.plotting`](src/mechatree/plotting/_mechanics.py) uses the function default.
- **Tests**: [tests/test_light.py](tests/test_light.py) adds `test_intercept_default_tau_attenuates_shadowed_leaf` (default 0.5 ⇒ shadowed leaf sees 0.5), `test_intercept_tau_zero_recovers_binary`, `test_intercept_tau_one_makes_leaves_transparent`, `test_intercept_tau_half_geometric_attenuation` (4-leaf stack ⇒ `[1, 1/2, 1/4, 1/8]`), and validation.   - **(b) Convergence criterion**: both `n_cut == 0` (primary, canonical) **and** ε-tolerance on canopy-mean (cheap early-exit), with iteration cap as safety net. The ε is the cheapest signal — at 1000-tree scale dropping 0.5 % of branches barely moves the pooled wind, so the loop exits at iter 2 without paying iter 2's prune sweep.
  - **(c) Per-tree vs forest-wide loop**: forest-wide. Per-tree would be incorrect under canopy-aware wind because pruning tree A changes the pooled `canopy_mean` that drives every tree's prune in the next sweep.

### Step 26 plan (momentum-wind overhaul — planned 2026-05-27)

**Why.** The screening-aware CFD wind (Step 25b/c) is the path to a Darwinian island tournament with mutual sheltering (Step 21 × 25). Three things block that being *right* and *fast*: (1) the name "actuator-disk" is wrong — there is no actuator disk, it's a momentum-conservation column march; (2) the model is physically inconsistent — pruning can use the per-branch screened field but **sensing still uses a uniform unit wind over 4 hardcoded angles**, so trees reinforce against a load they never feel; (3) the wind solve is the wall-clock bottleneck and several config knobs (`grid_size`, fixed-point iteration count) drive it. This arc fixes naming, removes a pointless mode, makes sensing and pruning consistent, and validates at island scale.

**Cross-cutting decisions already made (don't re-litigate):**
- **Per-branch is the point.** A 3-D solve averaged to one scalar == constant wind. The canopy-mean *wind mode* is deleted; per-branch is the only momentum behaviour. The mean survives only as the ε convergence thermometer.
- **`grid_size = 2` default.** It's the `O(branches) ≈ O(cells)` knee (coarsening further is branch-bound, ~no win) **and** it preserves the vertical + intra-crown wind gradient the model exists for. Afford the `n_angles` sensing sweep via **parallelism, not grid coarsening**.
- **Storm fixed within a generation.** Sample `(θ, amplitude)` once per gen; the fixed-point loop iterates *that* storm.
- **`U_uniform` is `U(z)` (inflow profile shape), not a storm distribution.** Storm strength per gen is `amplitude_cdf`; the two are orthogonal and compose.
- **Sensing == pruning loads.** The momentum-wind per-branch field is the single source of branch loads for both. Per-branch means *local sheltered wind per cell*, not "leaf drag" (leaf drag was always in both).

### Step 26 drag ½ρU² fix + Cauchy recalibration (2026-05-27)

Surfaced while explaining the ½ factor between `wind_force` and the CFD kernel: the **Fortran `wind_force` (mod_tree.f90:637) omitted the ½ρU² dynamic-pressure factor** (`force = V²·d·L·sin²θ`, no ½), folding it into the Cauchy calibration. The faithful port inherited the bug. Per the user (author), corrected for physical consistency with the momentum kernel (which already carries ½ρU²).

- **Fix**: restored the ½ in [`wind_force`](src/mechatree/_core/mechanics.cpp) (woody drag, where the ½ is physically required since D·L is geometric) **and** the leaf-cluster drag `S0·|U|·U` at all four sites (`calculate_stresses`, `prune`, and both stored-force variants — where the ½ is a convention the tuned `S0` absorbs). `C_D = 1`.
- **Why both**: applying the ½ uniformly to the *whole* wind load means the load scales by exactly ½, so **doubling the default `cauchy` (2.0e-5 → 4.0e-5)** reproduces the Fortran/Nat Commun reference **bit-for-bit** (½ load × 2× Cauchy = identical stress → identical Weibull → identical trees). Fixing only `wind_force` would have left *no* Cauchy able to reproduce the reference (woody is a minority of the load; leaf `S0` dominates and was unchanged).
- **Consequences**: the `default` model is unchanged in behaviour (reproduces the paper) but now physically correct internally. The `momentum` model becomes fully ½ρU²-consistent (woody + leaf) and shares the `cauchy = 4e-5` calibration. `examples/forest.yaml` + `tests/test_config.py` + `tests/test_mechanics.py` updated for the new Cauchy / the corrected `wind_force` value (`½·V²·d·L`). Champions (`data/S3_champions.json`) still apply (default reproduces the Fortran).

### Step 26d notes (closed 2026-05-27)

`Profile + island-scale validation`. The 26c sensing sweep added `n_sensing_angles` momentum solves per generation, so the first job was to find the bottleneck before optimising (per the user's standing "measure before building").

- **Profile** (R = 50, 239 trees, 35 k branches, realistic long-tail storm, `n_sensing_angles = 4`): 60 ms/gen; momentum solves are **66 % of wall-clock**; of those, **sensing is 68 %** (4.0 solves/gen) vs pruning's 1.9 fixed-point solves/gen. Mean 6.6 ms/solve. So the independent sensing solves are the target.
- **Parallelism measured, not built**: a `ThreadPoolExecutor` over the 4 solves runs at **0.67×** (slower) — the kernel's per-column Python loop holds the GIL, so threads only add contention; numpy's per-op GIL release isn't enough. A process pool would need per-generation pickling of the canopy arrays (and macOS uses `spawn`), so it only amortises at large scale. **Conclusion: cheap in-process fan-out doesn't pay; the real lever is a GIL-free Cython/OpenMP momentum kernel** (mirrors the Step-21b light-kernel port) — deferred to its own step, now justified by this measurement. Orthogonal lever: `n_sensing_angles` trades directional-sensing fidelity for ~linear sensing cost; tournament-scale runs can also fan out **forest-of-forests** (Step 21c) across independent replicates with no kernel change.
- **Feasibility (gate)**: momentum+sensing is ~linear in branch count (1.35 µs/branch at grid_size = 2). Scaling: R=30/18k → 21 ms/gen; R=50/44k → 50 ms/gen; R=80/90k → 121 ms/gen. Island projection (R = 200, ~400 k branches): **~0.5 s/gen → 1000 gens ≈ 9 min, 100k gens ≈ 15 h serial**. Tractable; the 100k-gen Nat Commun reproduction (Step 21c) is an overnight serial run, faster with the deferred kernel parallelism or by reducing `n_sensing_angles`.
- **Cold-start survival**: a screening-aware `run_tournament` (model = momentum, random founders, R = 30, 100 init, U=1 + long-tail storm, 50 gens) **survives and selects** — fills to carrying capacity (100→200), lineages winnow 100→20, biomass climbs. The earlier "random genomes + screening → mass extinction" worry doesn't bite at this storm level.
- **A/B (does screening reshape trees?)**: same seed/geometry, `default` (uniform) vs `momentum` (screened sensing+pruning) → 77 trees/15.6 k branches vs 84 trees/13.4 k branches. Resolving the screening measurably changes the stand.
- **Follow-ups (2026-05-27)**: (ii) ✅ notebook 07 §3 refreshed — the native/dendroflow comparison cells rewritten to the `momentum` model (canopy-mean demo + per-branch wake), TOC + mechanics + "what's next" prose updated, no removed-module imports left; (iii) ✅ `benchmarks/bench_momentum_wind_at_scale.py` instrumentation fixed (the old `_IterTracker` replaced `forest.wind_fn` with a plain callable, which broke the 26c `wind_fn.sense` path) — new `_WindTracker` proxy times sensing vs pruning solves separately and forwards `sense`/`sensing_angles`. (i) ⏳ GIL-free Cython/OpenMP kernel still the active next perf step.
- **`n_sensing_angles` default 4 → 2 (per user)**: each sensing angle is ~11–12 ms/gen (measured: 4→60, 2→36, 1→26 ms/gen at R=50/35k branches), so 2 ≈ halves the sensing cost. Angles stay random-per-gen draws from `angle_cdf`, so the directional load still integrates over generations (the user's framing). 2 orthogonal-ish directions are a coarser max-stress envelope than 4 — a fidelity/cost knob. Threads don't parallelise the solves (GIL-bound NumPy column-march, 0.67×), so the GIL-free Cython kernel is the route to running the remaining solves concurrently.

### Step 26e plan (GIL-free momentum kernel — planned 2026-05-27)

**Why.** Step 26d profiled the momentum step: the `n_sensing_angles` sensing solves dominate (68 % of the wind solves, wind = 66 % of wall-clock), and those solves are independent — but in-process parallelism doesn't pay because the kernel ([`_momentum_wind_kernel.py`](src/mechatree/wind/_momentum_wind_kernel.py)) is pure NumPy with a Python `for i in range(Nx)` column-march that holds the GIL (a thread pool over the 4 solves measured **0.67×**, i.e. slower). The fix is a compiled kernel that releases the GIL so the solves run concurrently. This is the justified next perf step; island-scale runs are already tractable serially (~0.5 s/gen at R = 200, 1000 gens ≈ 9 min), so this is an optimisation, not a blocker.

**Approach — test-driven, in this order** (mirrors the Step-21b [`light.cpp`](src/mechatree/_core/light.cpp) port, which OpenMP-parallelised over sun directions):

1. **Port one solve to a `nogil` C++/Cython function** `momentum_solve` — the per-cell drag (bincount → typed loop), the actuator-disk velocity update, the edge-clamped implicit cross-stream diffusion, and the per-branch `F_vec` / `U_branch` output. New `src/mechatree/_core/momentum.{h,cpp}` (or a `.pyx` with `nogil`), added to the `setup.py` Extension `sources`, with a Cython binding. Keep the Python-side pooling + grid build + rotation as-is (cheap; the column-march is the hot part).
2. **Equivalence test** vs the existing NumPy `compute_momentum_wind` — `atol ≈ 1e-10` on `U_out`, `F_vec_branch`, `U_branch` across a few grids/directions. This is the gate: the port must be numerically identical before any parallelism goes in. Add to [tests/test_wind_momentum.py](tests/test_wind_momentum.py).
3. **Parallelise the independent solves.** Two options, measure both: (a) a Python `ThreadPoolExecutor` over the `n_sensing_angles` angles — now effective because each solve runs `nogil`; (b) an OpenMP `parallel for` over angles inside one C++ entry point (the `light.cpp` pattern). The `setup.py` OpenMP probe + serial fallback from Step 21b is already in place.
4. **Re-measure**: the 26d thread-pool-vs-serial test (0.67× on NumPy) should flip to a real speedup; report island-scale ms/gen and the new 1000-/100k-gen projections.

**Notes / gotchas.**
- Keep the NumPy `compute_momentum_wind` as the reference implementation (the equivalence test needs it; it's also the readable spec).
- The kernel must stay bit-faithful to the ½ρU² + per-branch-force conventions from the Step-26 drag fix (woody `½ C_D D L U² sin²i`, the `F_vec` rotation back to world frame happens in the bridge, not the kernel).
- Pruning fixed-point solves are sequential (each depends on the previous cut), so only the sensing sweep parallelises; that's where 68 % of the wind time is anyway.
- Cheaper orthogonal lever already shipped: `n_sensing_angles` (default 2) trades directional fidelity for ~linear sensing cost; forest-of-forests (Step 21c) parallelises across independent tournament replicates with no kernel change.
